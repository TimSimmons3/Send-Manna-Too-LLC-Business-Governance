# Send Manna Too LLC Document Hierarchy Standard

Version: 0.1
Status: Approved
Owner: Send Manna Too LLC
Canonical Source: Markdown in Git repository
Applies To: All Send Manna Too LLC Business Governance Framework documents
Effective Date: 2026-07-08
Review Cycle: At least annually or upon material governance, business, legal, or operating-model change

## 2. Purpose

This standard defines the document hierarchy, document type definitions, placement rules, naming rules, conflict-resolution rules, and anti-duplication requirements for the Send Manna Too LLC Business Governance Framework.

The purpose is to ensure each framework document has one clear role, one correct location, one primary purpose, and a controlled relationship to every other framework document.

## 3. Scope

This standard applies to governance documents, advisory framework documents, standards, procedures, playbooks, templates, checklists, scripts, records, and other framework artifacts maintained by Send Manna Too LLC.

This standard does not define writing quality rules, publication formatting rules, logo requirements, confidentiality footer requirements, page-numbering requirements, or document review criteria. Those requirements belong in the Document Quality Standard or controlled publication templates.

## 4. Relationship to the Constitution

The Send Manna Too LLC Constitution is the highest internal governance authority for the framework.

This standard implements the Constitution by defining how documents are classified, placed, named, and controlled within the approved hierarchy.

If this standard conflicts with the Constitution, the Constitution controls unless a documented and approved exception states otherwise.

## 5. Document Hierarchy

The framework uses the following hierarchy:

1. Constitution
2. Policies
3. Standards
4. Frameworks
5. Procedures / Playbooks
6. Templates / Checklists / Scripts
7. Records

Higher-level documents govern lower-level documents. Lower-level documents must not override, weaken, or duplicate higher-level documents.

## 6. Document Type Definitions

### 6.1 Constitution

The Constitution establishes the highest-level governance principles, authority, hierarchy, and quality expectations for the framework.

There is only one Constitution.

### 6.2 Policies

Policies define mandatory management intent, risk posture, decision boundaries, and required outcomes.

Policies answer what must be true and why it matters. Policies should not contain detailed procedures, templates, or step-by-step instructions.

### 6.3 Standards

Standards define mandatory rules, minimum requirements, classification rules, quality expectations, or control requirements needed to implement policies and governance expectations consistently.

Standards answer what requirements must be met.

### 6.4 Frameworks

Frameworks organize related concepts, capabilities, operating models, or advisory methods into a coherent structure.

Frameworks answer how a topic is structured and how its parts relate to each other. Frameworks should not become procedures or templates.

### 6.5 Procedures / Playbooks

Procedures and playbooks define repeatable steps, decision paths, or execution guidance for a specific activity.

Procedures and playbooks answer how work is performed. They must align with the Constitution, applicable policies, standards, and frameworks.

### 6.6 Templates / Checklists / Scripts

Templates, checklists, and scripts provide reusable execution aids.

They support consistent delivery but do not create new governance requirements unless those requirements are already defined in a higher-level document.

### 6.7 Records

Records provide evidence that a decision, review, approval, exception, assessment, change, or activity occurred.

Records document what happened. Records should not define future requirements.

## 7. Document Placement Rules

Documents must be placed according to their primary purpose.

If a document defines governing principles or authority, it belongs in Foundation.

If a document defines mandatory management intent or risk posture, it belongs in the applicable policy location.

If a document defines mandatory requirements, classification rules, or quality rules, it belongs in the applicable standards location.

If a document organizes a domain, capability, or advisory method, it belongs in the applicable framework location.

If a document defines repeatable execution steps, it belongs in the applicable procedure or playbook location.

If a document is a reusable aid, it belongs in the applicable template, checklist, or script location.

If a document is evidence of action or approval, it belongs in the applicable record or release artifact location.

When placement is unclear, the document must be classified by primary purpose, not by title, convenience, or expected audience.

## 8. Document Naming Rules

Document names must be clear, durable, and purpose-specific.

Names should identify the firm, topic, and document type where practical.

Names must avoid vague labels such as general notes, miscellaneous guidance, final final, working copy, or temporary unless the document is intentionally temporary and clearly marked.

File names should use readable hyphen-separated words and the .md extension for canonical Markdown sources.

## 9. One Primary Purpose Rule

Each document must have exactly one primary purpose.

A document may reference related documents, but it must not absorb the purpose of another document type.

If a document begins serving multiple primary purposes, it must be split, reclassified, or revised.

## 10. No Duplicate Guidance Rule

Duplicate guidance is not permitted.

A lower-level document may summarize or reference higher-level guidance only when necessary for context.

When guidance is repeated, the canonical source must be identified and the duplicate text must be removed or converted to a reference.

## 11. Conflict Resolution

If two documents conflict, the higher-level document controls.

If two documents at the same level conflict, the more specific approved document controls unless a decision record states otherwise.

Conflicts must be documented and corrected through the appropriate decision, change, or exception process.

## 12. Document Lifecycle Alignment

Documents must follow the approved development lifecycle: planning, outline review, draft, technical review, executive review, red team review, editorial review, approval, Git commit, GitHub push, and baseline.

No document may bypass lifecycle steps unless an exception is documented, justified, and approved.

## 13. Exceptions

Exceptions to this standard must be documented, justified, and approved.

Exceptions must not compromise the Constitution, one source of truth, document traceability, non-duplication, or professional integrity.

Recurring exceptions may indicate that the hierarchy, placement rules, or document structure should be reviewed.
