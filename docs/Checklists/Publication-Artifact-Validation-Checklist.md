# Publication Artifact Validation Checklist

## Document Control

Document Status: Planned
Release: 0.3
Document Type: Checklist
Canonical Source: Markdown in Git

## Purpose

This checklist defines the validation controls used before approving Send Manna Too LLC Business Governance Framework publication artifacts.

The checklist applies to Word, PDF, and PowerPoint publication outputs and templates. It does not make publication artifacts authoritative sources.

## Core Validation Controls

| Control | Word | PDF | PowerPoint | Validation Requirement |
|---|---:|---:|---:|---|
| Source traceability | Yes | Yes | Yes | Artifact identifies or can be traced to approved Markdown source or approved release context. |
| Non-authoritative status | Yes | Yes | Yes | Artifact is clearly treated as a publication output, not the authoritative source. |
| Send Manna Too LLC branding | Yes | Yes | Yes | Branding, logo, or firm identification appears where technically supported. |
| Confidentiality marking | Yes | Yes | Yes | Property and Confidential appears where technically supported. |
| Page or slide numbering | Yes | Yes | Yes | Page or slide numbering appears where technically supported. |
| Professional format | Yes | Yes | Yes | Artifact is executive-ready, readable, and consistently formatted. |
| Header and footer rendering | Yes | Yes | Limited | Headers, footers, and markings render correctly where technically supported. |
| Placeholder review | Yes | Yes | Yes | Artifact contains no unresolved placeholder status markers before release. |
| Client-data review | Yes | Yes | Yes | Artifact contains no unintended client data or unnecessary confidential client information. |
| Source correction rule | Yes | Yes | Yes | Corrections are made in the approved source or controlled artifact source, then regenerated or re-exported. |

## Word Artifact Checklist

Before approving a Word publication artifact, confirm:

1. The document uses approved Send Manna Too LLC branding.
2. The header includes the logo where technically supported.
3. The footer includes approved firm identification where technically supported.
4. The footer includes Property and Confidential.
5. Page numbering follows Page 1 of X Pages where technically supported.
6. Document control information supports review, approval, traceability, and maintenance.
7. The output does not replace Markdown in Git as the authoritative source.

## PDF Artifact Checklist

Before approving a PDF publication artifact, confirm:

1. The PDF was generated from an approved Word, PowerPoint, Markdown, or controlled source artifact.
2. The PDF opens successfully.
3. The PDF content matches the approved source.
4. Headers and footers render correctly where technically supported.
5. Property and Confidential appears where technically supported.
6. Page numbering follows Page 1 of X Pages where technically supported.
7. The PDF contains no unintended client data.
8. The PDF contains no unresolved placeholder status markers.
9. The PDF does not replace Markdown in Git as the authoritative source.

## PowerPoint Artifact Checklist

Before approving a PowerPoint publication artifact, confirm:

1. The deck uses approved Send Manna Too LLC branding.
2. Logo or firm name placement is consistent.
3. Property and Confidential appears where technically supported.
4. Slide numbering appears where technically supported.
5. Heading, body, table, and footer styles are consistent.
6. The deck is concise, executive-ready, and traceable to approved source or release context.
7. The deck does not change the underlying approved meaning of controlled content.
8. The deck does not replace Markdown in Git as the authoritative source.

## Approval Rule

A publication artifact should not be approved unless the applicable checklist items are satisfied or an exception is documented in the decision log or future considerations register.
