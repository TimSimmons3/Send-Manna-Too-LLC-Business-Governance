## Purpose

This procedure defines the controlled process for producing publication-ready Word, PDF, and PowerPoint outputs from approved Send Manna Too LLC Business Governance Framework source content.

The purpose is to ensure that publication outputs are consistent, professional, traceable, and controlled without replacing Markdown in Git as the authoritative source of truth.

## Authority

Markdown in Git remains the authoritative source of truth for controlled governance framework content.

Word, PDF, and PowerPoint files are publication outputs unless a future approved decision changes the governance architecture.

## Scope

This procedure applies when creating controlled publication outputs for:

- Governance framework documents.
- Policies, standards, procedures, playbooks, templates, and checklists.
- Executive-ready reports.
- Client-ready advisory documents.
- Presentation materials derived from controlled governance content.

## Procedure Overview

The publication and export process follows these stages:

1. Confirm source readiness.
2. Confirm publication type.
3. Prepare publication output.
4. Validate formatting and content.
5. Record traceability.
6. Store or distribute the approved output.

## Step 1 - Confirm Source Readiness

Before creating a publication output, confirm that the source content is ready for publication.

Validate that:

- The source file is in the approved repository.
- The source file is controlled in Markdown where applicable.
- The source file has been reviewed for completeness.
- The source file has no unresolved placeholder status markers.
- The source file aligns with the Document Quality Standard.
- The source file does not duplicate controlled guidance from another document.
- The Git working tree status is understood before export.

## Step 2 - Confirm Publication Type

Select the appropriate publication output type based on the intended use.

| Output Type | Appropriate Use |
| --- | --- |
| Word | Formal editable publication format, review package, or client-ready document. |
| PDF | Final-form read-only distribution, archive, or evidence package. |
| PowerPoint | Executive presentation, board-style summary, training, or advisory briefing. |

Do not create publication outputs that are unnecessary, duplicative, or outside the approved release scope.

## Step 3 - Prepare Publication Output

Create the publication output using the applicable controlled requirements:

- Word outputs must align with the Word Publication Template Requirements.
- PDF outputs must align with the PDF Publication and Export Approach.
- PowerPoint outputs must align with the PowerPoint Publication Template Requirements.

Publication outputs should preserve the approved meaning of the source content.

Corrections should be made in the approved source first whenever technically practical.

## Step 4 - Validate Formatting and Content

Before approval or distribution, validate that the publication output:

- Opens successfully.
- Contains the expected document title.
- Contains Send Manna Too LLC branding where technically supported.
- Contains Property and Confidential where technically supported.
- Contains page or slide numbering where technically supported.
- Preserves the approved source meaning.
- Uses consistent formatting.
- Renders tables, lists, and headings correctly.
- Contains no unintended client data.
- Contains no unresolved placeholder status markers.
- Is appropriate for the intended audience.

## Step 5 - Record Traceability

Each controlled publication output should be traceable to its source.

Where practical, record:

- Source Markdown path.
- Source Git commit or release tag.
- Publication output path.
- Publication date.
- Export method.
- Reviewer or approver when applicable.

Traceability may be recorded in the release notes, change log, publication record, or approved evidence location.

## Step 6 - Store or Distribute the Approved Output

Store or distribute only validated publication outputs.

Approved outputs should be stored in an approved location such as:

- A controlled release package.
- An approved artifact folder.
- An approved client delivery folder.
- An approved backup location.

Do not distribute uncontrolled working copies as final deliverables.

## Naming Guidance

Publication output file names should be clear, stable, and traceable.

Recommended naming pattern:

`Send-Manna-Too-LLC-[Document-Name]-[Release-or-Version]-[YYYYMMDD].[extension]`

File names should avoid ambiguous labels, informal names, or uncontrolled version terms.

## Exception Handling

If a publication output cannot meet a required formatting or technical control, record the limitation and the compensating control.

Examples of acceptable compensating controls may include:

- Recording that a feature was not technically supported by the export tool.
- Preserving traceability through the release package.
- Correcting the approved source and regenerating the output.
- Deferring template automation to the Future Considerations Register.

## Restrictions

Publication outputs must not:

- Replace Markdown in Git as the authoritative source.
- Introduce uncontrolled changes to approved content.
- Add new governance decisions without updating the Decision Register.
- Include client-specific or confidential data unless approved for that purpose.
- Be treated as final until validation is complete.

## Validation Checklist

Before a publication output is approved, confirm:

- Source readiness was validated.
- Output type was appropriate.
- Required branding was included where technically supported.
- Required confidentiality marking was included where technically supported.
- Page or slide numbering was included where technically supported.
- Formatting was reviewed.
- Content was compared to the approved source.
- Traceability was recorded.
- Output location was approved.

## Future Implementation Note

This procedure defines the controlled publication and export process.

Specific automation scripts, Word templates, PDF tooling, or PowerPoint template files may be created later as controlled artifacts, but this Markdown procedure remains the controlled governance source.
