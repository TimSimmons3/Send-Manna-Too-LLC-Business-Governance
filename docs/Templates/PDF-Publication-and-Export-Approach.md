## Purpose

This document defines the controlled approach for producing PDF publication outputs for Send Manna Too LLC Business Governance Framework deliverables.

The purpose is to ensure that PDF outputs are professional, traceable, consistent, and clearly treated as publication artifacts rather than authoritative source records.

## Authority

Markdown in Git remains the authoritative source of truth for controlled governance framework content.

PDF files are final-form publication outputs. A PDF file must not become the primary record unless a future approved decision changes the governance architecture.

## Scope

This approach applies to PDF outputs created for:

- Governance framework documents.
- Executive-ready reports.
- Client-ready advisory documents.
- Policies, standards, procedures, playbooks, templates, and checklists.
- Release packages and formal publication records.

## PDF Role

A PDF output is a controlled rendering of approved source content.

PDF outputs may be used for:

- Executive distribution.
- Client distribution.
- Formal review packages.
- Archived release evidence.
- Read-only publication where formatting consistency is required.

PDF outputs must not be used to bypass Markdown review, Git version control, or approved release procedures.

## Approved Source Inputs

PDF outputs should be created from one of the following approved inputs:

- Canonical Markdown in Git.
- Approved Word publication output generated from canonical Markdown.
- Approved PowerPoint publication output generated from canonical Markdown or controlled presentation source.

Uncontrolled manual PDF edits should be avoided. If a PDF requires correction, correct the approved source and regenerate the PDF.

## Required PDF Elements

A controlled PDF output should include:

- Send Manna Too LLC branding where technically supported.
- Document title.
- Document owner.
- Version or release identifier.
- Publication date when applicable.
- Property and Confidential marking.
- Page numbering in the format Page 1 of X Pages where technically supported.
- Clear document structure.
- Consistent headings, tables, and spacing.

## Traceability Requirements

Each controlled PDF output should be traceable to its approved source.

At minimum, publication records should preserve:

- Source Markdown file path.
- Source Git commit or release tag when available.
- Intermediate publication file path when applicable.
- Export method.
- Export date.
- Reviewer or approver when applicable.

## Naming Requirements

PDF file names should be clear, stable, and traceable.

Recommended naming pattern:

`Send-Manna-Too-LLC-[Document-Name]-[Release-or-Version]-[YYYYMMDD].pdf`

File names should avoid:

- Ambiguous names.
- Informal labels.
- Client names unless the file is client-specific and approved for that purpose.
- Uncontrolled version terms.

## Export Controls

Before a PDF is released, confirm that:

- The source document has been reviewed and approved.
- The PDF was generated from an approved source.
- Required branding and confidentiality markings are present.
- Page numbering is present where technically supported.
- Formatting is readable.
- Tables render correctly.
- Headers and footers render correctly.
- Links work where applicable.
- The PDF contains no unintended client data.
- The PDF contains no unresolved placeholder status markers.

## Accessibility and Usability Expectations

PDF outputs should be readable and usable by the intended audience.

Where technically supported, PDF exports should preserve:

- Searchable text.
- Clickable links.
- Logical heading order.
- Readable tables.
- Sufficient contrast.
- Reasonable file size.

Scanned or image-only PDFs should be avoided unless there is a documented business reason.

## Restrictions

Controlled PDF outputs must not:

- Replace Markdown in Git as the authoritative source.
- Contain uncontrolled edits not reflected in the approved source.
- Introduce duplicate governance requirements.
- Contain confidential client information unless the PDF is explicitly approved for that client purpose.
- Be treated as editable source material.

## Validation Requirements

Before a PDF publication output is approved, validate that:

- The PDF opens successfully.
- The PDF content matches the approved source.
- Required markings are present.
- Page numbering is present where technically supported.
- Formatting is complete and readable.
- Tables and lists render correctly.
- No unintended sensitive or client-specific information is present.
- The output aligns with the Document Quality Standard.

## Future Implementation Note
